## 基本原則
- コードは常に日本語で分かりやすくコメントしてください。
- コメントは必ず終止形にしてください。
- コメントのですますは必ず終止形にしてください。。
- 全角括弧（「（」「）」）は半角括弧「()」に修正してください。
- 読みやすさ＞短さ。将来の読者（自分）を優先。
- 明示的に書け。暗黙より明示を選べ。
- 早めに返れ。ガード節でネストを浅く保て。
- 初心者にも理解できるよう専門用語を避け、必要なら簡単な説明を添えます。
- 質問が不明瞭な場合、その旨を丁寧に伝えます。
- コードの重複を避け、モジュール化を推奨します。
- データ主導で設計せよ。状態と遷移を明確にせよ
- 説明的な変数名を使用します（例：`is_loading`, `has_error`）。
- 必要に応じてROROパターンを使用します。
- 変更は小さいステップに分割し、それぞれを小さなテストで確認します。
- 新規コードを書く前に既存コードをレビューし、動作を理解します。
- 一貫性を守れ。同じ問題は同じパターンで解け。
- 運用・保守・監視などの懸念を常に意識します。
- セキュリティリスクを常に確認し、潜在的リスクがある場合は追加レビューを行います。
- 命名は「何をする/何であるか」が伝わる語を選べ
- 早口な短縮形を避けよ。ドメイン語彙を積極的に使え
- 引数は少なく保て。3つを超えたらオブジェクト化を検討せよ
- デフォルト引数に可変オブジェクトを使うな
- マジックナンバーを定数化せよ。列挙型を活用せよ
- コメントは「なぜ」を書け。「何」はコードに表現せよ
- 長い式は分解して名付けよ。中間変数で意図を見せよ


## コードのスタイルと構造

### Tech stack
- React

### Style
- 依存追加は理由を明記し、既存の選択肢で解決できない場合のみ。
- Secrets を埋め込まない（.env値やトークンを生成/貼り付けしない）。
- 関数コンポーネント + Hooks。クラスコンポーネントは作らない。
- 型境界（APIレスポンス/props/外部I/O）では明示型を置く。any禁止。
- UIはアクセシビリティ（role/label/keyboard）を満たす。
- Airbnb JavaScript Style Guide 準拠
- TypeScript 型定義必須 + TSDoc形式コメント
- 文字コード UTF-8
- 命名規約: camelCase / PascalCase / UPPER_SNAKE_CASE
- ディレクトリ名はケバブケースを使用します（例：data-processing）
- コンポーネント名は PascalCase を使用します（例：UserProfile）
- 関数名・変数名は camelCase を使用します（例：calculateTotal）
- 定数は UPPER_SNAKE_CASE を使用します（例：MAX_RETRY_COUNT）
- 型定義は常に明示します（any の使用は禁止）
- React コンポーネントは関数コンポーネントのみ使用します
- Props は必ず interface または type で定義します
- すべてのコンポーネント・関数に TSDoc コメントを記述します
- TSDoc は補足説明のみを含め、冗長な説明は書きません。
- コメント内の全角括弧（「（」「）」）は半角括弧「()」に統一します。
- useEffect は依存配列を必ず明示します。
- 状態管理はローカル優先（useState / useReducer）。グローバル状態は必要最小限。
- 副作用ロジックとUIロジックは分離します。
- API通信は services/ ディレクトリに分離します。

- エラー処理には明確なガード句を設け、早期リターンを推奨します。
- 明確でユーザーフレンドリーなエラーメッセージを返します。
- 入力は常にサニタイズし、インジェクション攻撃を防ぎます。
